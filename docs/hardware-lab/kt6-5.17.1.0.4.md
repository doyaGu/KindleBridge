# KT6 / 5.17.1.0.4 hardware bring-up evidence

- Date: 2026-07-17
- Host: Windows 11 x86-64
- Device target: `kindlehf`
Status: historical hardware evidence; composite enumeration and full-duplex
raw-bulk data remain valid, but the direct-controller recovery design is retired

> Lifecycle correction (2026-07-18): later cycles showed that direct configfs,
> `soft_connect`, and MTU3 rebind recovery can leave Windows with Code 43 or MTP
> without visible storage. Those results do not establish a safe production
> lifecycle. The current manager requires an unplugged cable and delegates both
> ownership transitions to stock `volumd`/HAL. Do not use this document as USB
> mode-switching instructions.

Device serial and product strings are intentionally redacted from this
open-source report.

## Runtime and storage

- Kernel: Linux `4.9.77-lab126`, ARMv7 hard-float, MT8110 Bellatrix.
- CPU: two Cortex-A53-class ARMv7 cores with NEON, VFPv4, AES, SHA1/SHA2, and
  CRC32 features advertised by the kernel.
- RAM: 993,636 KiB usable; no swap.
- glibc: 2.20. Cross-built probe requires at most GLIBC_2.18.
- `/`: read-only ext3.
- `/var/local`: writable ext4, about 415 MiB free during the probe.
- `/mnt/us`: `fsp` FUSE view backed by the ext4 loop image mounted at
  `/mnt/base-us`; the observed FUSE mount advertises `max_write=65536`.

## USB feasibility

- UDC: `11211000.usb`, configured at `high-speed`; its maximum is also
  `high-speed`.
- Runtime modules include `libcomposite`, `configfs`, `usb_f_fs`, `usb_f_rndis`,
  `usb_f_ecm`, `u_ether`, and `g_ether`.
- configfs is mounted and the stock `mtpgadget` contains `functions/ffs.mtp`.
- The stock gadget enables the Microsoft OS string (`MSFT100`) and config-level
  OS descriptors. The FunctionFS compatible ID was observed by Windows as
  `USB\MS_COMP_WINUSB`, which selected the inbox WinUSB driver. Its extended
  property did not create `DeviceInterfaceGUIDs`, including after the probe
  supplied a stable GUID descriptor. That negative result is consistent with
  the vendor FunctionFS OS-descriptor handling and is the behavior the design
  must accommodate.
- The inactive stock MTP composition reports VID `0x1949`, PID `0x9981`, USB
  2.0, device class/subclass/protocol zero, and Amazon/Kindle identity strings.
- During this run, the UDC was owned by the legacy single-function `g_ether`
  gadget. Windows enumerated it as `VID_0525:PID_A4A2` RNDIS, while the stock
  configfs MTP gadget was unbound (`UDC` empty, `connected=0`).

Conclusion: FunctionFS and configfs are present and usable, but there is only
one UDC. KindleBridge cannot attach a second independent gadget beside MTP or
`g_ether`; its raw-bulk interface must be part of the single active composite
configuration and share an explicit, recoverable USB-mode lifecycle. Raw bulk
can proceed. On this firmware, a development host must add the stable interface
GUID `{3F5EC011-3CD6-4E0D-819C-387BED7DB3B5}` to the enumerated interface; no
third-party kernel driver is required.

## KBP over current RNDIS baseline

The release ARM binary was copied to `/mnt/us`, run manually as a one-shot
allowlisted listener, and removed after testing. It never wrote rootfs or
installed a startup job. KBP used a 40-byte checked header and 1 MiB payloads.

| Test | Per direction | Aggregate | Integrity |
|---|---:|---:|---|
| 256 round trips, stop-and-wait | 15.01 MiB/s | 30.01 MiB/s | 512 MiB verified |
| 256 round trips, concurrent writer/reader | 14.39 MiB/s | 28.78 MiB/s | 512 MiB verified |

The current RNDIS path does not meet the raw-bulk median target of 30 MiB/s in

## Composite enumeration result

The timed FunctionFS cycle successfully replaced the legacy `g_ether` gadget
with the stock configfs MTP function plus one KindleBridge function. Windows
enumerated all three nodes using inbox drivers:

- composite parent: `VID_1949:PID_9981`;
- stock MTP: interface `MI_00`, operational with the inbox MTP driver;
- KindleBridge: interface `MI_01`, class/subclass/protocol `ff/4b/01`, matched
  to the inbox WinUSB driver through `USB\\MS_COMP_WINUSB`.

The compatible ID selected WinUSB, but no `DeviceInterfaceGUIDs` value was
installed, so `nusb` initially had no device-interface path. Supplying the MS
OS 1.0 extended property from FunctionFS did not change that result. The lab
host therefore has the stable GUID
`{3F5EC011-3CD6-4E0D-819C-387BED7DB3B5}` configured manually for this interface.
This is acceptable for bring-up but not yet the final open-source onboarding
experience.

Recovery testing found two distinct cases. A `soft_connect` disconnect/connect
works when `g_ether` remains the active gadget. After switching from configfs
back to `g_ether`, the MTU3 controller can retain stale VBUS state; rebinding
the MTU3 platform driver refreshes it and restored RNDIS in about four seconds
in the isolated hardware test. The integrated recovery requires UDC state
`configured`, performs bounded soft reconnect and module retries, then uses
that controller rebind. One earlier integrated failure cycle recovered RNDIS
automatically.

The first attempt to restore as soon as the data probe exited used `kill -0`.
An exited zombie still satisfies that check, so the watchdog waited for its
timeout. The probe now writes an explicit completion marker and the watchdog
listens to that marker; the corrected immediate path still needs one hardware
cycle.

## Raw-bulk data result

The first 1 MiB transfer stalled with the exact MTU3 diagnostic
`req length > supported MAX:65532 requested:1048576`. FunctionFS endpoint I/O
is now capped to 63 KiB per kernel request while KBP retains 1 MiB logical
frames. A regression test verifies that a 1 MiB round trip never asks the
underlying endpoint for more than 63 KiB.

With that fix, 128 1 MiB KBP payloads in each direction completed concurrently
in 8.540 seconds with integrity checks:

| Mode | Host to device | Device to host | Aggregate |
|---|---:|---:|---:|
| Full duplex | 14.99 MiB/s | 14.99 MiB/s | 29.97 MiB/s |

This is a shared high-speed USB 2.0 bus result. It does not establish the
30 MiB/s one-direction target. Dedicated host-to-device and device-to-host
modes are implemented and await hardware measurement.

## Formal bridge and superseded development-package result

The original standalone MRPI/KUAL development package was installed on the KT6. It
stores its payload below `/mnt/us/kindlebridge`, state below
`/var/local/kindlebridge`, and writes no stock rootfs path. The package contains
no USBNetLite or KindleRoot integration. A pre-existing RNDIS session was used
only as a one-time rescue channel during bring-up; the installed manager itself
recorded and restored generic `g_ether` or stock MTP state. That lifecycle
manager has been superseded; only the transport and enumeration observations in
this section remain evidence for the current implementation.

On 2026-07-21, the superseded layout failed deterministically after a fresh
install: Windows completed HELLO and one `device list`, then the device logged
`heartbeat write delayed: Stale file handle` and the launcher reported the same
`ESTALE` failure. The long-lived A/B control plane was still below the FSP/MTP
userstore. The replacement layout moves the launcher, daemon slots, manifests,
 PID files, and heartbeat to `/var/local/kindlebridge/control`; the installer
 performs a one-time atomic migration from the old location only while the old
 manager is inactive, and leaves the new package inactive. Hardware revalidation
 of the migrated layout is a
required follow-up, so the earlier repeated-call result must not be treated as
covering this package revision.

Windows enumerated the formal daemon as the stock MTP `MI_00` plus the
KindleBridge WinUSB `MI_01`. Separate short-lived host CLI invocations all
completed over the same bound FunctionFS gadget: `device list`, feature
negotiation (`exec.v1`, `sync.v1`), root `shell -c`, and a two-command
line-oriented shell session. `sync push` then transferred a revised USB manager
through KindleBridge itself, and the root shell activated it.

The KT6 sometimes left the MTU3 controller bound without presenting a usable
device after a configfs transition. One such transition produced a transient
Windows unknown-device notification. A superseded recovery experiment checked
for UDC `configured`, attempted bounded `soft_connect`, and finally rebound the
MTU3 platform driver before rebinding the configfs gadget. In a controlled test with
no cable replug, an MTP -> KindleBridge -> MTP -> KindleBridge cycle returned
the composite parent, MTP interface, and KindleBridge interface with Windows
PnP status OK; `device list` and shell remained operational afterward. Later
failures falsified the assumption that this was a generally safe recovery path.

## Stock-volumd manager and watchdog validation (2026-07-18)

The stock-`volumd` ownership manager and A/B launcher were installed from the
KT6 package whose SHA-256 was
`EE289A71ABB45694430BA5E2C93D9EC5E3E0DD5C477E62A988915470DD48BCF4`.
The package was copied through stock MTP and read back with the same hash before
MRPI installation.

An earlier build repeatedly lost its USB gadget shortly after a successful
enumeration. A timestamped device monitor showed that the daemon PID changed

and termination after a sustained 1,001 ms gap.

## Bulk backpressure and interrupted-session recovery (2026-07-18)

Sustained 32 MiB pushes exposed three independent failure layers. First, the
host filled the OUT endpoint without draining the device's connection and
stream credit frames. The FunctionFS IN side then starved, so the device could
neither publish more credit nor continue reading OUT. The host now drains the
initial credit window and waits for both windows to be replenished after each
4 MiB batch.

After that fix, transfers progressed farther but still stopped at varying
offsets. The kernel recorded `kindlebridged: page allocation failure: order:4`.
The previous 63 KiB FunctionFS operations required sixteen physically
contiguous 4 KiB pages, which became unreliable on the fragmented production
kernel even when tens of MiB remained free. Device and host USB operations are
now capped at 16 KiB. Three consecutive 32 MiB pushes and one 32 MiB pull then
completed with matching SHA-256, and the kernel recorded no new allocation
failure from the revised daemon.

A final fault injection killed both host processes 300 ms into a push, while
the device was inside a 64 KiB frame payload. Windows kept the USB interface
configured, so FunctionFS delivered no disconnect or HUP to delimit the stale
session. Protocol v2 therefore gives every host/device hello exchange a fresh
128-bit session ID. Recovery uses a two-second timeout only for the initial USB
handshake, reopens the interface, concurrently drains stale device-to-host data,
sends one 1 MiB host-to-device resynchronisation burst followed by a valid ping
frame, and ignores stale inbound frames until the matching fresh device hello
arrives. The 1 MiB recoverable host frame limit is independent of the 16 MiB
connection credit window. Application traffic retains indefinite USB
backpressure rather than acquiring a wall-clock timeout.

The device also records the start of each push DATA frame. If a session ends
while receiving that frame, it truncates the staging file to that boundary and
recomputes the prefix hash before publishing resumable metadata. This prevents
resynchronisation padding from becoming durable file data and retransmits at
most one frame.

Follow-up fault injection exposed three more implementation errors. First, after
recovering the byte boundary, the device's active-session loop continued to
read the old OPEN exchange. It interpreted the recovery ping as an OPEN frame,
ended `serve_session`, closed the FunctionFS endpoints, and waited for another
ENABLE event that Windows did not owe it. The outer USB state machine now
returns directly to the hello gate after every recovered top-level read error;
a regression test verifies that the ping marker is consumed only after a fresh
hello.

Second, the FunctionFS resynchroniser issued a `read_exact` operation for every
discarded byte. It now scans buffered endpoint chunks with `fill_buf`/`consume`.
An attempted host workaround using `WinUsb_ResetPipe` was also removed: phase
instrumentation showed that the reset itself consumed 53.735 seconds on this
composite device. Recovery now opens the existing vendor interface normally and
does not reset, reselect, or reconfigure either the vendor interface or the MTP
composite parent.

Third, diagnostic pulls using 4 KiB DATA payloads stopped after increments of
exactly 4 MiB minus 4 KiB. The FunctionFS writer had submitted the 40-byte KBP
header and the 4,096-byte payload as separate USB write sequences. The payload
ended on a 512-byte high-speed packet boundary, leaving the host's 16 KiB read
pending at the end of a credit batch. FunctionFS now submits the header and
payload as one contiguous frame, so the 4,136-byte sequence ends in a short
packet. A request-boundary regression test protects this property.

The final hardware regression used daemon SHA-256
`874F20BFB0293CFE1700CC2B4CD3511B368684652602E9A1C471F2AC1778AA8B`
from slot A:

| Check | Result |
|---|---|
| Host processes killed during 32 MiB pull | CLI and server both terminated |
| Interrupted host staging | 1,429,504 bytes preserved |
| Recovery without replug or reboot | Online in 2,447 ms total |
| Recovery phase timing | Initial timeout 2,006 ms; reopen 2 ms; resync exchange 408 ms; matching hello 0 ms |
| Resume | 32 MiB complete in 1.94 s, SHA-256 match |

## Sustained-sync watchdog and FSP backing-store correction (2026-07-19)

A 128 MiB push exposed a second, distinct launcher failure. Production slot
manifests allowed only 1,500 ms between heartbeats, while the heartbeat itself
lived on `/mnt/us`. A sustained userstore write and final `fsync` could stall
that FUSE path longer than 1.5 seconds. The launcher then sent `SIGTERM` to a
healthy daemon, which closed FunctionFS and made the whole USB composite
disappear from Windows. Kernel and launcher evidence showed a controlled daemon
termination followed immediately by a respawn; there was no SIGILL, segfault,
OOM, page-allocation failure, or MTU3 fault.

Production manifests now use 10,000 ms for both the steady-state heartbeat
timeout and the healthy-after interval. The watchdog logs the exact timeout
reason before restarting a daemon. With the fixed launcher and unchanged NEON
daemon, three consecutive 128 MiB pushes completed without a PID change or USB
loss. The bridge remained `active`, slot B remained selected, and the log had no
new heartbeat-triggered restart.

The same run isolated the remaining push bottleneck. During a formal push,
`kindlebridged` used about 38--52% CPU, `fsp` used about 57--70%, system time was
commonly 50--75%, and I/O wait stayed near zero. The mount layout explained the
profile: `/mnt/us` is the `fsp` FUSE view with 64 KiB maximum writes, while
`/mnt/base-us` is the ext4 backing mount. Controlled 128 MiB `dd + fsync`
measurements were 13.62 MiB/s through `/mnt/us` and 43.57 MiB/s through
`/mnt/base-us`. A file created through the backing mount was immediately visible
through `/mnt/us` with the same size.

The USB manager therefore selects `/mnt/base-us/kindlebridge-data` only when
`/proc/mounts` positively identifies both `/mnt/us` as `fuse.fsp` and
`/mnt/base-us` as a mounted backing path. Other layouts explicitly fall back to
`/mnt/us/kindlebridge-data`. The logical user-visible tree, same-volume staging,
atomic rename, path confinement, and BLAKE3 verification are unchanged.

Final formal-sync measurements used the same 128 MiB source for every run:

|---|---|
| Push 1 | 7.259 s, 17.63 MiB/s |
| Push 2 | 7.392 s, 17.32 MiB/s |
| Push 3 | 7.617 s, 16.80 MiB/s |
| Push median | 17.32 MiB/s |
| Pre-fix push median | 9.63 MiB/s |
| Median improvement | 79.8% |
| Pull | 7.141 s, 17.93 MiB/s |
| Integrity | Source and pull SHA-256 both `254BCC3FC4F27172636DF4BF32DE9F107F620D559B20D760197E452B97453917` |
| Stability | Same daemon PID across all pushes, pull, and final status |
| Manager | `active`, `link=configured`, slot B, timeout disabled |

This closes the known FSP-path performance regression and the sustained-write
watchdog failure. It does not waive the broader high-speed gate: the safe
16 KiB FunctionFS/WinUSB path still needs one-direction profiling and further
work before a 30 MiB/s claim is justified.

## USB mode cycles and sleep-resume finding (2026-07-19)

The stock-`volumd` manager subsequently completed repeated Bridge-to-MTP
handbacks and MTP-to-Bridge re-entry on the KT6 without direct controller writes
or a reboot. A long sleep then exposed a separate scheduler-order race: on wake,
the launcher observed the existing heartbeat as 26,538,782 ms old before the
daemon heartbeat thread ran and restarted an otherwise healthy daemon.

The launcher now treats heartbeat content as an observed progress token instead
of trusting its cross-process wall-clock age alone. If an already-healthy
launcher misses more than one heartbeat window, it grants exactly one normal
window for heartbeat progress to resume; sustained lack of progress still trips
the watchdog. Deterministic tests cover both recovery and eventual termination.
Repeated physical sleep/wake remains required before closing the hardware gate.

## Shell v2 and connection-actor validation (2026-07-21)

The installed `0.1.0-dev.17` package was built for `kindlehf`, required at most
GLIBC 2.18, and had SHA-256
`592003A959E917A0B327C5BF80B131667AB20A37D0A94F1EEFD91B4A5D351982`.
The daemon reported the same version on the device and executed as root.

Four independent defects were closed before the final run. A scheduler queue
mutation hidden inside `debug_assert_eq!` disappeared from release builds; the
FunctionFS bulk writer unnecessarily depended on the ep0 control lock; the USB
actor reopened endpoints for every invalid recovery-fill header and could lose
the following HELLO; and late host shell input after the device had closed that
stream was treated as a session-fatal unknown stream. The last case exhausted
the remaining connection credit and froze an unrelated sync stream. Terminal
stream DATA is now consumed only within its already-bounded residual credit,
discarded, and returns connection credit without reopening the stream. Release
tests cover the scheduler path, and actor tests cover recovery-fill/HELLO and
late shell input isolation.

The default sync block was reduced from 1 MiB to 256 KiB because a USB DATA frame
is non-preemptible. This preserved bulk throughput while allowing the weighted
scheduler to service interactive traffic between frames.

| Hardware check | Result |
|---|---|
| Persistent PTY | cwd/environment persistence, fd 0/1/2 TTY, 30x120 resize, Ctrl-C, and remote exit status passed |
| Raw shell | stdout/stderr separation, NUL data, exit 37, and 32 MiB output without truncation passed |
| Shell closed during 128 MiB push | shell exit 19 preserved; push exit 0 in 6.04 s |
| Two shells + push + log | 120 echoes: P50 15.77 ms, P95 25.42 ms, max 70.68 ms |
| Mid-push shell close | first shell exit 21; second shell remained usable and exited 22; push completed in 6.09 s |
| Shared host service reclaim | three consecutive stop/automatic-start cycles returned online without replug; stop settled in 148--149 ms |

These results close the known shell/sync starvation and normal host-server
reclaim failures. Physical unplug, long sleep/wake, and longer concurrent soak
remain release gates.

## dev.29/dev.30 FunctionFS recovery correction (2026-07-22)

The daemon-restart experiments disproved the proposed inherited-control-fd
recovery design. FunctionFS creates its endpoint files after descriptors and
strings are written to `ep0`, disables the function when all files close, and
unregisters the gadget when the first function closes its endpoints. ADB keeps
`ep0`, bulk-out, and bulk-in owned and serviced by the same long-lived `adbd`
process; its reconnect loop reopens bulk endpoints without replacing that
process. These are lifecycle constraints, not a missing UDC retry. See the
[kernel FunctionFS documentation](https://docs.kernel.org/usb/functionfs.html)
and [AOSP usb_ffs.cpp](https://android.googlesource.com/platform/system/core/+/c745f09b33804242c43f823242f0112645ed3a98/adb/daemon/usb_ffs.cpp).

Dev.30 attempted to keep only `ep0` open in a separate monitor. On the KT6 this
prevented the replacement daemon from opening `ep0` (`EBUSY`) and development
mode rolled back with `E-OTHER`. The package was not accepted. The experiment
has been removed completely.

The corrected dev.31 policy does not claim transparent recovery. A health
monitor observes the launcher, daemon identity, heartbeat, endpoints, UDC, and
gadget binding without opening an endpoint or writing USB state. If the daemon
identity changes, status becomes `degraded` with `reason=daemon-restarted`.
KUAL then instructs the user to unplug USB, choose file transfer, and choose
development mode. No automatic UDC unbind, HAL transition, driver operation, or
controller reset occurs. Transparent backend replacement remains a future
architecture change requiring a long-lived USB frontend that owns and services
all FunctionFS endpoints and proxies KBP over IPC.

## dev.32 terminal-stream lifecycle correction (2026-07-22)

The dev.31 concurrency follow-up reproduced a host-session stall without any
device, UDC, or driver failure. Three local clients were started at the same
instant: a 32 MiB push, repeated short raw Shells, and 20 log RPCs. The log
calls completed, while sync and Shell remained blocked for more than 30
seconds. The remote file had already reached its complete 32 MiB size, the
gadget remained `configured`, and the daemon and health monitor remained live.
Restarting only the host server recovered the session.

The connection actor retained every terminal stream in its worker registry.
After a worker consumed `CLOSE` or `RESET`, another receive on that stream could
wait forever, and connection-credit processing continued scanning all
historical short-lived streams. The existing fairness test opened its Shells
before starting sync/log traffic, so it did not lock down terminal retirement.

Dev.32 retires only the affected actor stream after a terminal frame is
consumed or successfully written. It wakes that stream's remaining open,
receive, send, and write waiters with `Disconnected`; the wire state continues
to retain the terminal protocol state, so late authorized DATA still consumes
connection credit without reopening the stream or damaging another stream. A
new regression test first failed because a second receive timed out, then
passed with the fix while a subsequent unrelated stream opened normally.

With the fixed host server, the exact simultaneous-client reproducer completed
a 32 MiB push in 1.86 seconds. After another 200 Shell and 200 log short-stream
cycles, the same gate completed the push in 1.85 seconds while all 20 Shell and
20 log operations returned. Server handles and threads remained stable. The
full Rust workspace and warning-denied Clippy gate passed.
